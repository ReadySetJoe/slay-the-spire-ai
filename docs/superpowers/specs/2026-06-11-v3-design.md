# Slay the Spire AI — v3 Design

**Date:** 2026-06-11
**Branch:** `v3`
**Status:** Approved, ready for implementation

---

## Goal

Improve the learning algorithm so the agent discovers optimal play patterns naturally — card ordering within a turn, energy efficiency, relic acquisition, and enemy-intent-aware blocking — without hard-coded heuristics. Introduce a dynamic card synergy system that improves as the agent plays more runs. Add a hung-episode watchdog so program freezes are tracked and recovered from automatically.

V2 entry points (`main.py --v2`) are preserved unchanged. V3 is invoked via `main.py --v3`.

---

## Architecture Overview

| Component | V2 | V3 |
|---|---|---|
| Policy | `MaskablePPO` (flat MLP) | `MaskableRecurrentPPO` (LSTM) |
| Observation | 227 features (3 blocks) | 250 features (4 blocks) |
| Combat intent encoding | 1 float per monster | 3 binary flags per monster + 2 aggregates |
| Turn context | None | 12 features (new Block 4) |
| Card synergy | Static heuristics | CardScorer EMA (self-updating) |
| Relic rewards | Weak (0.0–0.05) | Strong (0.15–0.25) |
| Energy waste penalty | −0.3 | −0.5 |

| Entry point | `--v2` | `--v3` |
| Source | `src/v2/` | `src/v3/` |

**What is unchanged:** action space (104 actions, same masking rules), episode structure (one run = one episode), `RunRewardShaper` terminal/combat reward formulas (except energy penalty weight and relic rewards), all v1 and v2 files.

## Observation Space — 250 Features

Four fixed-size blocks. Non-active blocks are zero-padded each step. Exact feature indices defined in `run_encoder.py`.

### Block 1 — Global (55 features, unchanged from v2)

Identical to v2. See v2 design doc for full breakdown.

### Block 2 — Combat (122 features, redesigned intent encoding)

| Group | V2 features | V3 features | Change |
|---|---|---|---|
| Hand | 70 | 70 | Unchanged |
| Monsters | 30 (5 × 6) | 40 (5 × 8) | Intent float → 3 binary flags (+10) |
| Aggregate intent | 0 | 2 | New: `any_enemy_attacking`, `attacking_count/5` (+2) |
| Player powers | 5 | 5 | Unchanged |
| Turn metadata | 3 | 3 | Unchanged |
| Debuff signal | 3 | 3 | Unchanged |
| Padding | 1 | — | Removed (−1) |

**Total combat block: 123 features** (was 112; net change = +10 intent flags + 2 aggregates − 1 padding = +11).

#### Per-monster features (8 per monster, 40 total)

| Offset | Feature | Description |
|---|---|---|
| 0 | `hp_ratio` | `current_hp / max_hp` |
| 1 | `max_hp_norm` | `max_hp / 400` |
| 2 | `block_norm` | `block / max_hp` |
| 3 | `is_attacking` | Binary: intent in `{ATTACK, ATTACK_BUFF, ATTACK_DEBUFF, ATTACK_DEFEND}` |
| 4 | `is_buffing` | Binary: intent in `{BUFF, DEFEND_BUFF, ATTACK_BUFF}` |
| 5 | `is_debuffing` | Binary: intent in `{DEBUFF, STRONG_DEBUFF, ATTACK_DEBUFF}` |
| 6 | `vulnerable_stacks` | `vulnerable / 10` |
| 7 | `weak_stacks` | `weak / 10` |

This replaces the single `intent_encoded` float. Three independent binary flags allow the agent to learn clean logical rules (e.g., "all `is_attacking` = 0 → skip block cards") without needing to learn an arbitrary numeric threshold.

#### Aggregate intent features (2 features, after monster block)

| Feature | Encoding | Purpose |
|---|---|---|
| `any_enemy_attacking` | Binary: 1 if any live monster has `is_attacking = 1` | The primary "should I block?" signal |
| `attacking_enemy_count` | `count / 5` | Severity — one attacker vs. three is different |

These aggregates are redundant with the per-monster flags but give the MLP head of the LSTM an easy linear path to the key blocking decision without needing to aggregate five monster slots internally.

### Block 3 — Non-combat (60 features, unchanged from v2)

Identical to v2. See v2 design doc for full breakdown.

### Block 4 — Turn Context (12 features, new)

Active only during combat. Zero-padded on all non-combat screens. Reset at the start of each new turn (END action) and at the start of each new combat.

| Offset | Feature | Encoding | What it captures |
|---|---|---|---|
| 0 | `turn_position` | actions taken this turn / 10 | How early in the turn we are |
| 1 | `energy_spent_this_turn` | energy spent / 4 | How committed we are to this turn |
| 2 | `attacks_played_this_turn` | count / 5 | Whether we've been dealing damage |
| 3 | `skills_played_this_turn` | count / 5 | Whether we've used setup / utility |
| 4 | `powers_played_this_turn` | count / 3 | Whether buffs are active |
| 5 | `strength_gained_this_turn` | stacks / 10 | Flex / buff payoff available for remaining attacks |
| 6 | `vulnerable_applied_this_turn` | Binary | Whether Bash landed before Strike |
| 7 | `weak_applied_this_turn` | Binary | Whether weak debuff is active |
| 8 | `damage_dealt_this_turn` | cumulative / max_hp | Offensive commitment this turn |
| 9 | `block_gained_this_turn` | cumulative / max_hp | Defensive commitment this turn |
| 10 | `last_card_was_buff` | Binary | Last played card was a power / buff |
| 11 | `last_card_was_debuff` | Binary | Last played card applied vulnerable or weak |

**Why this enables ordering learning:** when the agent plays Flex (a buff), features 4, 5, and 10 activate. On the next step, when it plays Strike and deals more damage due to strength, those features are present in the observation the LSTM associates with high reward. Over many episodes the LSTM learns: `last_card_was_buff = 1` + `strength_gained > 0` → play attacks now. And upstream: seeing Flex in hand at `turn_position = 0` → play it first to create that state.

## Turn State Tracking in V3RunEnv

`V3RunEnv` subclasses `RunEnv` (from `src/v2/run_env.py`). It maintains `self._turn_state: dict` and overrides `reset()` and `step()`.

### Turn state dict

```python
{
    "actions_taken": 0,
    "energy_spent": 0,
    "attacks_played": 0,
    "skills_played": 0,
    "powers_played": 0,
    "strength_gained": 0,
    "vulnerable_applied": False,
    "weak_applied": False,
    "damage_dealt": 0.0,
    "block_gained": 0.0,
    "last_card_was_buff": False,
    "last_card_was_debuff": False,
}
```

### Update logic (after each combat step)

After sending the command and receiving the new state, for any combat action (action < 91 on a `NONE` screen):

```
energy_spent    += prev_state.energy - new_state.energy
damage_dealt    += sum(max(prev_hp[i] - new_hp[i], 0) for each monster)
block_gained    += max(new_state.player_block - prev_state.player_block, 0)
strength_gained += max(new_strength_power - prev_strength_power, 0)
vulnerable_applied |= any monster gained vulnerable stacks
weak_applied       |= any monster gained weak stacks
actions_taken   += 1
```

For card plays specifically (action 0–59): look up the card at that slot in `prev_state.hand`, increment `attacks_played` / `skills_played` / `powers_played` based on `card.type`, and set `last_card_was_buff` / `last_card_was_debuff` from `card_properties`.

### Reset triggers

| Event | Action |
|---|---|
| `reset()` called | Full reset to zeros |
| Action == 60 (END) | Full reset — new turn |
| `prev_state.screen_type != "NONE"` and `new_state.screen_type == "NONE"` | Full reset — new combat |

## Reward Design

### Changes from v2

**Relic rewards — strengthened:**

| Context | V2 | V3 |
|---|---|---|
| Chest relic (OPEN) | `0.0` | `+0.25` |
| Boss relic pick | `+0.10` (added in v2 hotfix) | `+0.20` |
| Combat reward relic | `+0.05` | `+0.15` |
| Shop relic | `+0.05` | `+0.15` |

Rationale: relics are permanent upgrades with run-wide impact. A chest relic at floor 5 has expected value comparable to winning several future combats. The reward must be large enough to dominate the noise of exploration; `+0.25` is meaningful relative to the `[-1.0, +2.0]` terminal reward scale.

Card reward tier values are unchanged (`tier_value × 0.05`, max 0.05). This means the agent will prefer relics over most cards at card reward screens, which is correct behavior for Ironclad A0. An S-tier card (0.05) is competitive with a weak relic (0.15) only when synergy score pushes it up.

**Energy waste penalty — increased:**

```
# On END action (action 60):
energy_waste_penalty = −0.5 × (energy_remaining / max_energy)
```

Was `−0.3`, now `−0.5`. Ending a turn with 2 of 3 energy unspent costs `−0.33` — comparable to taking a meaningful hit. The turn context block's `energy_spent_this_turn` feature gives the LSTM direct visibility into how much energy has been used, so it can learn the correlation between full-energy turns and high reward very quickly.

### Unchanged from v2

Terminal reward, combat step reward (damage dealt/taken, kills, debuff gain), debuff-before-damage bonus, non-combat rewards (card pick, rest, smith, map). See v2 design doc.

## CardScorer EMA

Dynamic, self-updating card synergy scores that replace the static hand-crafted heuristics in `run_encoder.py`.

### How it works

`CardScorer` maintains a score per `card_id` in `[0.0, 1.0]`. After each combat resolves (detected by `NONE` → non-combat screen transition), it receives the list of card IDs played in that combat (accumulated by `V3RunEnv` in `self._combat_cards_played: list[str]`, reset on each new combat entry) and a `performance_signal` derived from the combat outcome:

```
performance_signal = clamp(total_damage_dealt / total_enemy_max_hp, 0.0, 1.0)
```

where `total_damage_dealt` is cumulative monster HP lost during that combat and `total_enemy_max_hp` is the sum of all enemies' max HP. This is always in `[0.0, 1.0]` — a combat where you dealt 100% of available HP (e.g., killed all enemies efficiently) scores 1.0; a combat where you dealt little damage scores low. It captures offensive efficiency without being confounded by the player's own HP loss.

EMA update applied to every card ID in `combat_cards_played`:

```
score[card_id] = (1 − α) × score[card_id] + α × performance_signal
```

- `α = 0.05` (slow update — smooths over ~20 combats before a card's score stabilizes)
- Initial score for all cards: `0.5` (neutral, inherits from static tier list on first encounter)
- Cards never played retain their initial score

Scores persist to `data/card_scores.json` and reload on startup. Format:

```json
{
  "card_scores": {
    "Inflame": 0.82,
    "Strike_R": 0.41,
    "Bash": 0.71
  },
  "total_combats_scored": 14820
}
```

### Integration with encoder

`V3RunEncoder.encode()` accepts a `card_scorer: CardScorer` argument (passed through from `V3RunEnv`). In `_encode_noncombat()`, the `synergy_score` field of each choice (Block 3, offset 1 per choice) uses `card_scorer.score(card_id)` instead of the static `_synergy()` heuristic.

During the first few thousand episodes before scores converge, the initial `0.5` neutral score provides a reasonable fallback. After ~10k combats the scores reflect real performance data.

### Class interface

```python
class CardScorer:
    def __init__(self, path: str = "data/card_scores.json", alpha: float = 0.05): ...
    def score(self, card_id: str) -> float: ...
    def update(self, cards_played: list[str], performance_signal: float) -> None: ...
    def save(self) -> None: ...  # atomic write to path
    def load(self) -> None: ...
```

`V3RunEnv` calls `card_scorer.update()` and `card_scorer.save()` at each combat-end transition. Save is atomic (write to `.tmp`, rename) to prevent corruption if the process is killed mid-write.

## Hung Episode Watchdog

### Problem

`communicator.receive_state()` blocks indefinitely if the game or CommunicationMod freezes. Training stalls, the run is not tracked, and manual intervention is required.

### Design

`V3RunEnv` wraps every `receive_state()` call with a threading-based timeout (signal-based timeouts are not available on Windows):

```python
class HungEpisodeError(Exception): ...

def _receive_with_timeout(self) -> GameState:
    result, error = [None], [None]

    def _recv():
        try:
            result[0] = self.communicator.receive_state()
        except Exception as e:
            error[0] = e

    t = threading.Thread(target=_recv, daemon=True)
    t.start()
    t.join(self._timeout_seconds)

    if t.is_alive():
        raise HungEpisodeError(f"No game response after {self._timeout_seconds}s")
    if error[0]:
        raise error[0]
    return result[0]
```

`V3RunEnv.step()` catches `HungEpisodeError` and returns:

```python
obs = self.encoder.encode(self._current_state, self._turn_state)
return obs, 0.0, False, True, {"hung": True, "floor": self._current_state.floor}
```

`terminated=False, truncated=True` — Gymnasium convention for "episode ended for external reasons." PPO bootstraps the return from the value function rather than treating this as a death. The agent receives no reward and no penalty — it simply isn't held responsible for a program bug.

`V3RunEnv.reset()` also wraps its `receive_state()` calls. If a hung error occurs during reset, it re-raises (the training loop must handle environment reset failures — SB3 will retry).

`timeout_seconds` defaults to `20.0`, configurable at construction.

### Tracking

`RunTracker` gains a `hung_count: int = 0` field. `EpisodeLoggerCallback` increments it when `info.get("hung")` is `True` and logs:

```
[V3] HUNG EPISODE at floor {floor} — program failure, not counted as death
```

The dashboard shows a "Hung" stat chip alongside Wins / Deaths.

## Model & Entry Point

```python
# main.py --v3
from sb3_contrib import MaskableRecurrentPPO  # requires sb3-contrib >= 2.3.0
from src.v3.run_env import V3RunEnv
from src.v3.card_scorer import CardScorer

card_scorer = CardScorer(path="data/card_scores.json")
env = V3RunEnv(
    communicator=communicator,
    run_tracker=tracker,
    card_scorer=card_scorer,
    timeout_seconds=20.0,
    energy_waste_penalty=0.5,
)

model = MaskableRecurrentPPO(
    "MlpLstmPolicy",
    env,
    verbose=1,
    n_steps=512,
    batch_size=64,
    n_epochs=10,
    gamma=0.99,
    gae_lambda=0.95,
    learning_rate=3e-4,
    lstm_hidden_size=256,
    tensorboard_log="data/v3_tensorboard/",
)
```

`n_steps=512` is chosen to collect roughly one full run per rollout (STS runs average 200–500 steps). The LSTM requires episode-aware rollout collection, which `MaskableRecurrentPPO` handles internally via `_last_episode_starts`.

**Masking compatibility:** verify at implementation time that the installed sb3-contrib version exposes `MaskableRecurrentPPO`. If not available, implement a thin shim that overrides `MaskablePPO`'s policy class with an LSTM actor-critic. `RunActionSpace.get_action_mask()` is unchanged either way.

Checkpoint path: `data/v3_checkpoints/`. Existing callbacks (`EpisodeLoggerCallback`, checkpoint callback) carry over unchanged.

## Out of Scope for V3

- Multi-character support (Silent, Defect, Watcher)
- Ascension levels above 0
- Act 4 / The Heart
- Curriculum learning or self-play
- Hyperparameter search (manual tuning only)
- Card score visualization in the dashboard (scores are in JSON, not surfaced in UI)
