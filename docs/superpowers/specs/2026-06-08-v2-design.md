# Slay the Spire AI — v2 Engine Design

**Date:** 2026-06-08  
**Branch:** `v2`  
**Status:** Approved, ready for implementation

---

## Goal

Replace the v1 split architecture (rule-based `SimpleAgent` + separate `CombatEnv`) with a unified RL agent that controls every game decision — combat, potions, card rewards, shop, map, events, and rest — across a single full-run episode. V1 entry points (`main.py` bare and `main.py --rl`) are preserved unchanged. V2 is invoked via `main.py --v2`.

---

## Architecture Overview

```
main.py --v2
  └── RunEnv(communicator, run_tracker)
        ├── reset()   → send START, advance past title screen, return encoded obs
        └── step(action) → translate action → command, send, receive state,
                           compute shaped reward, return (obs, reward, done, info)
```

### New files

```
src/v2/
  __init__.py
  run_env.py           # RunEnv(gym.Env) — one episode = one full run
  run_encoder.py       # GlobalEncoder + per-screen feature blocks → flat obs vector
  run_action_space.py  # RunActionSpace — 103-action fixed space + screen masking
  run_reward.py        # RunRewardShaper — per-decision reward signals
```

### V1 preservation

`src/agent.py`, `src/combat_env.py`, `src/game_loop.py`, and all v1 support files are untouched. `SimpleAgent` is not used by v2.

## Episode Boundary

- **Start:** `reset()` sends `START IRONCLAD 0` and waits for the first actionable state.
- **End:** GAME_OVER screen. Episode terminates with reward `+2.0` (win) or `-1.0` (death).
- One episode = one complete run from floor 1 to either the boss kill or death.

---

## Action Space

Fixed 104-action discrete space. All actions are always defined; screen-specific masking zeroes out invalid actions before the policy samples.

| Index range | Count | Command produced | Active on screens |
|---|---|---|---|
| 0–9 | 10 | `PLAY {slot+1}` | Combat (no-target cards) |
| 10–59 | 50 | `PLAY {slot+1} {target}` | Combat (targeted cards) |
| 60 | 1 | `END` | Combat |
| 61–65 | 5 | `POTION Use {slot}` | Combat (no-target potions) |
| 66–90 | 25 | `POTION Use {slot} {target}` | Combat (targeted potions) |
| 91–98 | 8 | `CHOOSE {0–7}` | Card reward, shop, map, event, grid, boss reward |
| 99 | 1 | `PROCEED` | Any screen that has PROCEED |
| 100 | 1 | `PURGE` | Shop |
| 101 | 1 | `CHOOSE rest` | Rest |
| 102 | 1 | `CHOOSE smith` | Rest |
| 103 | 1 | `OPEN` | Chest |

**Masking rules:**
- During combat: 0–90 enabled per playability/target validity; 91–103 disabled.
- During CARD_REWARD: 91–93 (CHOOSE 0–2, or 0–3 with relic) + 99 enabled; all others disabled.
- During SHOP_SCREEN: 91–97 (up to 5 cards + 3 relics) + 99 + 100 enabled per stock/gold; others disabled.
- During MAP: 91–93 enabled per available nodes; others disabled.
- During REST: 101–102 enabled; others disabled.
- During EVENT, GRID, HAND_SELECT, COMBAT_REWARD, BOSS_REWARD: appropriate CHOOSE range + PROCEED.
- During CHEST: 103 (OPEN) enabled; all others disabled.

## Observation Space

Flat vector of **227 features** divided into three fixed-size blocks. Non-active blocks are zero-padded each step. Exact feature indices are defined in `run_encoder.py`; this doc specifies group sizes and semantics.

### Block 1 — Global (55 features, always present)

| Group | Features | Description |
|---|---|---|
| Player stats | 8 | `hp_ratio, max_hp/400, floor/55, gold/999, act/3, ascension/20, energy/4, block/max_hp` |
| Deck composition | 8 | total/60, attack%, skill%, power%, curse_status%, exhaust_card_count/10, strength_synergy_count/5, draw_synergy_count/5 |
| Potions | 20 | 5 slots × `[has_potion, is_healing, is_attack, requires_target]` |
| Relics | 7 | relic_count/20, 6 binary flags: Akabeko, Anchor, Burning Blood, Centennial Puzzle, Philosopher's Stone, Astrolabe |
| Screen type | 12 | One-hot: NONE(combat), CARD_REWARD, REST, MAP, CHEST, EVENT, SHOP_ROOM, SHOP_SCREEN, GRID, HAND_SELECT, COMBAT_REWARD, BOSS_REWARD |

### Block 2 — Combat (112 features, non-zero only when `is_in_combat`)

| Group | Features | Description |
|---|---|---|
| Hand | 70 | 10 cards × 7 features `[cost/5, type_encoded, is_playable, applies_vulnerable, applies_weak, draws_cards, gains_block]` |
| Monsters | 30 | 5 × 6 features `[hp_ratio, max_hp/400, block/max_hp, intent_encoded, vulnerable/10, weak/10]` |
| Player powers | 5 | `strength/10, dexterity/10, weak_stacks/5, vulnerable_stacks/5, barricade_flag` |
| Turn metadata | 3 | `draw_pile_count/60, discard_pile_count/60, turn_number/20` |
| Debuff signal | 3 | `debuff_card_fraction_in_hand, avg_damage_potential, vuln_multiplier_available` |
| Padding | 1 | Reserved |

### Block 3 — Non-combat screen (60 features, non-zero only when not in combat)

| Group | Features | Description |
|---|---|---|
| Choices | 32 | Up to 8 choices × `[tier_value/1.0, synergy_score/1.0, cost_ratio, is_available]` |
| Deck synergy context | 5 | `exhaust_synergy, strength_synergy, draw_synergy, block_synergy, curse_removal_value` |
| Screen metadata | 8 | `gold_after_purchase/999, hp_heal_ratio, node_elite_flag, node_rest_flag, node_shop_flag, node_treasure_flag, node_event_flag, node_monster_flag` |
| Padding | 15 | Reserved for v2.1 expansion |

## Reward Design

All rewards are bounded in `[-1.0, +2.0]`. Combat steps dominate training signal early; non-combat shaping nudges strategic decisions.

### Terminal rewards

Terminal reward is always floor-scaled so that dying deeper into a run is meaningfully better than dying early. This provides gradient signal throughout training, not just on wins.

```
terminal_reward = (floor / 55) * 3.0 - 1.0
```

Range: `-1.0` (floor 0) → `+0.5` (floor 27) → `+2.0` (floor 55, win).

A win (Act 3 boss killed) naturally produces the maximum `+2.0` since it occurs at floor 55. No separate win bonus is needed — the formula handles it.

| Example outcome | Floor | Terminal reward |
|---|---|---|
| Death on floor 1 | 1 | −0.95 |
| Death on floor 10 | 10 | −0.45 |
| Death on floor 27 | 27 | +0.47 |
| Death on floor 40 | 40 | +1.18 |
| Win (floor 55) | 55 | +2.00 |

### Combat step rewards

```
reward = damage_dealt/max_hp
       - damage_taken/max_hp
       + 0.1 × monster_kills
       + 0.05 × max(new_debuff_stacks - prev_debuff_stacks, 0)
       - 0.3 × energy_wasted_fraction   # only on END action
```

Where `energy_wasted_fraction = energy_remaining / max_energy` when action is END. The penalty is intentionally large (-0.3 max) so the agent learns quickly that ending a turn with unused energy is bad. Ending with 1 of 3 energy wasted costs -0.1 — comparable to taking a meaningful hit.

**Debuff-before-damage bonus:** `+0.03` applied on the *damage* step when:
- A vulnerable or weak card was played earlier this turn (tracked in step state)
- The damage card targets an enemy that now has > 0 stacks of that debuff
- The enemy had 0 stacks at the start of the turn

This rewards ordering — applying Bash *before* Strike, not after.

### Potion rewards

Potion use is a combat step. The standard combat delta reward applies. No additional bonus/penalty — the agent learns timing from outcomes.

### Non-combat rewards

| Screen | Action | Reward |
|---|---|---|
| CARD_REWARD | Pick card | `tier_value(card) × 0.05 + synergy_score(card, deck) × 0.05` |
| CARD_REWARD | PROCEED (skip) | `0.0` |
| SHOP_SCREEN | Buy card (S/A tier) | `tier_value × 0.05 + synergy_score × 0.05 - cost_ratio × 0.02` |
| SHOP_SCREEN | Buy relic | `+0.05` flat |
| SHOP_SCREEN | PURGE (remove card) | `+0.03` if removed card is curse/status/D-tier, else `0.0` |
| SHOP_SCREEN | PROCEED | `0.0` |
| REST | Heal | `hp_gained / max_hp × 0.2` |
| REST | Smith | `+0.05` flat |
| MAP | Any choice | `0.0` (delayed signal from subsequent screens) |
| EVENT | Any choice | `0.0` (outcome reflected in HP/gold changes on next state) |
| CHEST | Open | `0.0` (relic value captured in global relic features) |

### Synergy score

`RunRewardShaper.synergy_score(card_id, deck)` returns a value in `[0.0, 1.0]`:

1. Get the card's property flags (`applies_vulnerable`, `draws_cards`, `gains_block`, etc.) from `card_properties.py`.
2. Cross-reference against deck composition signals:
   - If deck has ≥ 3 exhaust cards and picked card benefits from exhaust → `+0.3`
   - If deck has ≥ 3 strength-scaling cards and picked card scales with strength → `+0.3`
   - If deck has ≥ 3 draw cards and picked card is a power → `+0.2`
   - If picked card applies vulnerable and deck has ≥ 3 high-damage attacks → `+0.3`
3. Clamp to `[0.0, 1.0]`.

These thresholds are initial heuristics. The RL agent's gradient still drives overall strategy — synergy scoring just gives a weak shaping signal at pick time.

## Screen Handling

Every `step()` goes through the RL agent. `RunEnv` translates the action index to a CommunicationMod command string using `RunActionSpace.action_to_command(action, state)`.

`RunEnv.step()` never silently auto-handles a screen. If a screen type is unrecognized, the action mask exposes only `PROCEED` (index 99), and the agent always has a valid escape.

## Out of Scope for V2

- Multi-character support (Silent, Defect, Watcher) — Ironclad only
- Ascension levels above 0
- Act 4 / The Heart
- Relic-specific reward shaping beyond the 6 binary flags
- Learned synergy scores (CardScorer-style EMA for card pairs) — deferred to v2.1
